§1 Executive Summary

navanVoyageAI is a corporate travel multi-agent AI platform built as a CMU Capstone demonstration. It combines large language model orchestration, Model Context Protocol (MCP) tool sidecars, real-time streaming, a complete booking workflow, and a twelve-tab administrator console — all deployed as seven Docker services behind an nginx reverse proxy.

The platform answers a straightforward business question: can a conversational AI system replace a traditional corporate travel portal without sacrificing policy compliance, auditability, or operational visibility? navanVoyageAI's answer is yes — by decomposing the travel workflow into six collaborating AI agents that each own a specific domain (search, policy, destination intelligence, booking, evaluation, and orchestration), and by surfacing every LLM decision through structured SSE event streams that feed directly into real-time admin dashboards.

Key Result: A traveller can type "I need a business class flight from JFK to London next Tuesday, a hotel in Mayfair, and a mid-size car" in a single message and receive a fully parallel response — flight options, hotel cards, car results, UK visa advisory, weather, and a corporate policy verdict — all within one streaming turn, with every agent action logged and scored by an asynchronous judge.

§2 Project Background & Motivation

Problem Statement

Corporate travel management is a domain where policy compliance, cost control, and traveller experience are perpetually in tension. Legacy travel booking portals present employees with long search forms, opaque policy gates, and disconnected approval workflows. When a traveller wants to book a complex multi-leg international trip with hotel and car, they typically interact with three or four separate systems, consult an HR policy document, and submit to a manual approval chain that introduces days of latency.

The navanVoyageAI project asks: what if a conversational AI assistant understood corporate travel policy as deeply as a seasoned travel manager, could search flights, hotels, and cars in parallel, and explained policy decisions in plain language — all in a single chat turn?

Academic Context

navanVoyageAI is built as the CMU Capstone demonstration for the MSIT program, class of 2026. The capstone brief required a platform that:

  • Demonstrates a complete multi-agent AI system with at least four cooperating agents
  • Integrates with external tool APIs via a structured protocol
  • Shows real-time LLM evaluation and observability
  • Provides a production-credible deployment story (Docker, nginx, CI)
  • Is demonstrable in a 20-minute live demo session

Relationship to EnterpriseDataAgent

navanVoyageAI is a standalone sibling project to the EnterpriseDataAgent platform. It reuses the base multi-agent architecture patterns (BaseAgent ABC, AgentContext/AgentResult contracts, SSE streaming pipeline) but is a fully independent codebase targeting the travel domain. The two repos share no runtime dependencies — navanVoyageAI uses its own MongoDB database (nva_db), its own Docker network (nva-net), and its own port range (5210–5211, 4210, 8100–8103, 27117).

Design Principles

Separation of Concerns

Each agent owns exactly one domain. The OrchestratorAgent routes but never searches; the PolicyAgent enforces rules but never fetches prices; the JudgeAgent evaluates but never responds to the user. This makes each agent independently testable and replaceable.

Deterministic Policy

Policy checks are deterministic Python — no LLM call, no ambiguity. A business class ticket either violates the $3,000 price cap or it does not. This design ensures auditability and eliminates hallucinated policy decisions.

Full Observability

Every MCP tool call, every agent token, every latency measurement, every policy verdict, and every judge score is captured and surfaced through the admin console. Nothing happens in a black box.

Demo-First UX

The platform is designed around five specific demo scenarios that can each be completed in under three minutes. The UI surfaces the most visually compelling moments — parallel agent badges, MCP tool call cards, Tree of Thought branch comparison, booking wizard steps — to communicate architectural sophistication to a non-technical audience.

Request Flow — Chat Turn

A single chat message traverses the following path from browser to LLM and back:

  1. POST /api/chat/message — JWT-authenticated request lands at the FastAPI backend. The request body carries conversation_id, message, and optional session_context.
  2. SSE Response Stream opens — the backend switches immediately to an EventSourceResponse. The client begins receiving events before any agent has produced a token.
  3. Short-Term Memory load — the last 5 conversation turns are retrieved from nva_conversations and summarised by Claude Haiku into a compact context block.
  4. Long-Term Memory retrieval — any stored user preferences (preferred airline, seat class, frequent flyer) are fetched from nva_user_memory and prepended to the Orchestrator prompt.
  5. OrchestratorAgent — classifies the intent, selects which downstream agents to invoke, and (for complex queries) runs Tree of Thought to decide the best response strategy. Emits agent_start and token SSE events.
  6. Parallel dispatch — SearchAgent, PolicyAgent, and DestinationAgent run concurrently via asyncio.gather. Each emits its own agent_start, MCP tool call events, and agent_done events.
  7. MCP tool calls — SearchAgent opens FastMCP Client connections to the three sidecars, calls tools, and receives structured JSON responses. Each call emits mcp_tool_call and mcp_tool_result SSE events that feed the MCP Inspector panel.
  8. BookingAgent — activates only when a booking confirmation is requested; generates NVA reference numbers and writes to nva_bookings.
  9. Response aggregation — the Orchestrator collects all agent results, merges them into a single coherent reply, and streams it token-by-token to the client.
  10. JudgeAgent (async) — fires after the agent_done event; evaluates the complete exchange asynchronously; writes a 5-criteria score to nva_eval_scores. Does not block the user-visible response.
  11. Audit logging — every agent action, tool call, and policy verdict is written to nva_audit_log for later review in the Admin console.

§4 Multi-Agent Framework

Agent Design Philosophy

Every agent in navanVoyageAI inherits from BaseAgent, a Python abstract base class that enforces a common interface: each agent receives an AgentContext (message, conversation history, session entities, user profile) and returns an AgentResult (content string, metadata dict, agent name). This contract allows the Orchestrator to treat all agents uniformly while each agent implements its own domain logic in the run() method.

OrchestratorAgent

The OrchestratorAgent is the entry point for every conversation turn. Its responsibilities are:

  • Intent classification — determines whether the query requires flight search, hotel search, car rental, booking actions, destination information, or a combination thereof.
  • Complexity gate — if the query scores above the complexity threshold (multiple destinations, constraint conflicts, itinerary optimisation), it triggers the Tree of Thought sub-strategy before dispatching.
  • Parallel dispatch — launches SearchAgent, PolicyAgent, and DestinationAgent concurrently using asyncio.gather; the combined wall-clock time is the slowest agent, not their sum.
  • Response synthesis — receives all AgentResults, merges the content sections, applies the session context, and streams the final response to the SSE consumer.
  • Token streaming — alone among agents, the Orchestrator uses streaming LLM calls to emit token SSE events that drive the animated typing indicator in the UI.

SearchAgent

The SearchAgent operates in two phases. First, it calls an LLM (gpt-5.4, temp=0.1) with a structured extraction prompt to parse the user message into a typed parameter object: origin IATA, destination IATA, depart date, cabin class, hotel city, check-in/check-out, car city, pickup/return dates, and vehicle class. Second, it calls the appropriate MCP sidecars in parallel based on which parameter groups are populated (search_flights if origin+destination+date present; search_hotels if search_hotels=true; search_cars if search_cars=true).

PolicyAgent

The PolicyAgent is deliberately LLM-free. It receives the SearchAgent's raw results and applies a deterministic Python rule engine against the six loaded corporate policies. Each policy defines hard limits for flight price, hotel nightly rate, car daily rate, cabin class, hotel tier, and car class. The agent returns a PolicyVerdict with three fields: compliant: bool, violations: list[str], and suggestions: list[str]. Violations are surfaced in the UI as amber warning badges on the relevant search result cards.

DestinationAgent

The DestinationAgent generates a structured JSON briefing for each destination city in the query. The briefing covers: visa requirements (by employee nationality from the employee record), current weather season and typical temperature range, local timezone and UTC offset, currency and rough exchange rate, cultural tips relevant to business travel, and any active travel advisories. The agent uses gpt-5.4 at temperature 0.5 to allow varied but accurate descriptions.

BookingAgent

The BookingAgent handles create and cancel operations for flight bookings. It generates a globally unique NVA-YYYY-NNNNN reference number, writes the booking record to nva_bookings, emits an agent_start SSE event, and returns a confirmation payload that the UI uses to display the booking success screen. The BookingAgent does not call MCP tools for booking creation — it treats the pre-searched flight data as authoritative and writes directly to MongoDB, reflecting the mock nature of the Amadeus dataset.

JudgeAgent

The JudgeAgent runs asynchronously after the user-facing response is complete. It is invoked as a fire-and-forget coroutine so it never adds latency to the turn. It receives the full exchange (user message + assistant response + agent metadata) and calls an LLM with a structured evaluation prompt that produces five numeric scores (1–5) and a brief rationale for each. The scores are written to nva_eval_scores and surfaced in the Eval Metrics admin tab.

§5 Tree of Thought Reasoning

Motivation

Standard chain-of-thought prompting works well for straightforward queries ("book a flight from JFK to London"). For complex, multi-constraint queries — "optimise a three-leg executive itinerary for the lowest cost that still meets policy, avoids a visa issue in transit, and returns before the Friday board meeting" — a single reasoning chain frequently settles on a locally optimal solution that misses globally better alternatives.

Tree of Thought (ToT) addresses this by generating multiple independent reasoning branches, evaluating each branch against the problem constraints, and selecting the highest-scoring branch as the final strategy before dispatching agents. This is implemented in backend/application/agent/tot_strategy.py.

Complexity Gate

The OrchestratorAgent runs a lightweight heuristic (_is_complex()) before deciding whether to invoke ToT. The gate activates if any of the following conditions are true:

  • The message contains three or more distinct travel components (flight + hotel + car + destination brief)
  • The message references a budget constraint alongside a policy tier constraint
  • The message involves multi-city or multi-leg routing
  • The message contains explicit optimisation language ("cheapest", "best", "lowest cost that still", "prioritise")
  • The Orchestrator's intent classification returns confidence < 0.6 on the primary intent

Branch Generation

When ToT is activated, the OrchestratorAgent calls the LLM three times in parallel with a specially designed branch-generation prompt that instructs each call to take a different strategic angle:

  • Branch 0 — Cost-first: minimise total travel spend within policy limits
  • Branch 1 — Quality-first: maximise comfort and convenience within policy limits
  • Branch 2 — Policy-first: strictly satisfy every policy rule, then optimise

Each branch produces a structured JSON object containing: the proposed search parameters, the reasoning chain, predicted policy compliance, estimated cost tier, and a self-assessment confidence score (0–1).

Branch Evaluation

Each branch is scored by a second LLM pass (or by the deterministic PolicyAgent when the branch has enough parameter detail to evaluate). The scoring rubric has four dimensions:

DimensionWeightDescription
Policy compliance40%Does the proposed itinerary satisfy all active policy rules?
Cost efficiency25%How close is the total cost to the minimum possible within policy?
Traveller preference alignment20%How well does the strategy match the user's stored preferences?
Completeness15%Does the strategy address all components of the request?

Branch Selection and Execution

The highest-scoring branch is selected. Its parameter set is passed to the agent dispatch pipeline (SearchAgent, PolicyAgent, DestinationAgent) as if it were the directly extracted parameters from a simple query. The Orchestrator logs the branch selection decision, including the scores of the rejected branches, in the conversation metadata for display in the MCP Inspector's ToT section.

SSE Visualisation

The ToT process is fully visible in the MCP Inspector panel. The panel renders a ToTReasoningCard component that shows:

  • A "Tree of Thought activated" header with the complexity reason
  • Three collapsible branch cards, each showing the branch strategy name, reasoning text, and evaluation score
  • A gold highlight on the selected branch
  • The total ToT latency (typically 1.5–3 seconds for three parallel branch generations)

Performance Characteristics

Latency budget: ToT adds approximately 1.5–3 seconds to complex queries. The three branch-generation calls run in parallel (asyncio.gather), so the overhead is the slowest single branch call plus the evaluation pass — not three sequential calls. For simple queries the gate does not activate and there is zero ToT overhead.

Example: Complex Query

"I need the cheapest business-class flight from SFO to Tokyo with a 4-star hotel and a full-size car — but it has to fit within my executive travel policy and I need to be back by the 15th."

This query triggers the ToT gate because it has five components (flight + hotel + car + policy constraint + return date constraint) and explicit cost optimisation language. The three branches explore: (0) cheapest available business class options with minimum hotel tier; (1) premium options at the tier upper limit; (2) the most policy-safe options closest to the per-diem. In testing, Branch 0 wins ~65% of the time when cost is explicitly mentioned; Branch 2 wins ~25% when the user profile indicates a compliance-sensitive department.

§6 Model Context Protocol (MCP) Architecture

What is MCP?

The Model Context Protocol (MCP) is a standardised JSON-RPC–like transport specification that allows LLM applications to call external tools in a provider-agnostic way. Instead of hardcoding HTTP calls to vendor APIs inside the agent code, the agent opens a connection to an MCP server, discovers its available tools via the /tools endpoint, and calls them by name with typed arguments. The MCP server is responsible for translating the tool call into the underlying data operation.

navanVoyageAI uses FastMCP 3.4.2, the Python implementation of the MCP specification. Each sidecar is a FastMCP server that exposes its tools via the streamable_http_app() transport, which wraps the MCP protocol in a standard ASGI application served by uvicorn. The SearchAgent connects to sidecars using the FastMCP Client class over HTTP.

§7 Memory Architecture

Overview

navanVoyageAI implements a two-tier memory architecture that allows the system to maintain conversational coherence within a session (short-term) while accumulating personalised preferences across sessions (long-term). Both tiers feed into the Orchestrator's system prompt at the start of each turn, ensuring that every agent response is informed by what the system knows about the user.

Short-Term Memory (STM)
Sliding window of the last 5 conversation turns. Summarised by Claude Haiku (fast, cheap) into a compact context block. Lost when the conversation session ends. Lives in MongoDB nva_conversations for retrieval but is summarised fresh each turn.
Long-Term Memory (LTM)
Persistent user preference store in nva_user_memory. Updated after each session with extracted preferences (airline, seat class, dietary requirements, frequent flyer numbers). Loaded at the start of every new session.

Short-Term Memory — ShortTermMemory

The ShortTermMemory class (backend/application/agent/memory.py) implements a sliding context window with the following parameters:

  • Window size: 5 turns (10 messages — 5 user + 5 assistant)
  • Max characters: 2,000 characters after summarisation
  • Summariser model: Claude Haiku (claude-haiku-4-5) — chosen for speed and low cost
  • Entity extraction: Haiku extracts a session_context dict containing named entities (airports, dates, hotels, car classes) that persist across the window for parameter resolution in subsequent turns

§8 Corporate Policy Engine

Design: Deterministic, LLM-Free

The PolicyAgent is deliberately implemented without an LLM call. Corporate travel policy is a binary domain: a $3,200 business class ticket either exceeds the $3,000 policy cap or it does not. Introducing LLM inference into this decision creates the risk of hallucinated thresholds, inconsistent verdicts, and unauditable compliance claims. By using a deterministic Python rule engine, navanVoyageAI guarantees that the same search result produces the same policy verdict every time — a property essential for any compliance-adjacent application.

Policy Rule Evaluation Logic

The PolicyAgent receives the full set of SearchAgent results (flights, hotels, cars) and the employee's travel tier from their profile. It evaluates each result against the applicable policy and produces a PolicyVerdict:

class PolicyVerdict:
    compliant: bool
    violations: list[str]      # e.g. ["Flight $3,200 exceeds $3,000 Business cap"]
    suggestions: list[str]     # e.g. ["Consider Economy Plus at $1,850 (within policy)"]
    applicable_policy: str     # e.g. "Executive"
    checked_items: list[dict]  # per-item compliance flags

Rules evaluated per flight result:

  • Price in USD ≤ policy max_flight_price
  • Cabin class ≤ policy max_cabin_class (Economy < Economy Plus < Business < First)
  • Days until departure ≥ policy min_advance_days

Rules evaluated per hotel result:

  • Nightly rate in USD ≤ policy max_hotel_nightly
  • Star tier ≤ policy max_hotel_tier

Rules evaluated per car result:

  • Daily rate in USD ≤ policy max_car_daily
  • Vehicle class ≤ policy max_car_class (Economy < Compact < Mid-size < Full-size < SUV < Luxury)

Policy UI Integration

Policy verdicts are surfaced in the UI at two levels:

  • Search result cards: each FlightCard, HotelCard, and CarCard displays a green "Policy OK" badge or an amber "Policy Warning: [violation]" badge below the price.
  • Booking wizard (Step 3 — Review): a policy checkbox must be acknowledged before the Confirm Booking button activates. If the selected itinerary has any policy warnings, they are listed explicitly in the review step.

Policy Admin CRUD

The Admin → Policy tab renders a full CRUD interface for the six policies. Each policy can be viewed as a table row, clicked to open an edit modal with field-level inputs for every limit and class cap, and saved back to MongoDB. Changes are validated client-side (numeric ranges, enum class values) before the API call. This is intended to simulate the HR/finance admin workflow where policies are periodically updated to reflect new cost guidelines or organisational changes.

Per-Diem Note

The policy engine records a per_diem_usd field on each policy (Standard: $75/day, International: $95/day, Executive: $150/day, Company Exec: $250/day) but per-diem is not enforced against search results — it is provided to the DestinationAgent as context for its cost briefing ("Your per diem of $150/day covers most restaurant meals in London; fine dining will require expense approval").

§10 Booking System

BookingWizard — 4-Step Modal

Clicking Select → on any FlightCard opens the BookingWizard — a centred modal dialog with a four-step progressive form. The wizard persists state across steps using a useBookingWizard hook that holds the complete booking draft. The modal has a 48px header with the flight summary (carrier, route, price), a step indicator bar, and a scrollable body.

Step 1 — Passenger Details

Collects: full name, date of birth (date picker), passport number, passport expiry, email address, phone number, and emergency contact. Fields are pre-populated from the authenticated user's employee record where available (name, email). Client-side validation runs on blur; the Next button is disabled until all required fields pass.

Step 2 — Seat & Meal Preference

Presents a seat preference selector (Window / Middle / Aisle, with Exit Row and Extra Legroom checkboxes) and a meal preference dropdown (Standard, Vegetarian, Vegan, Halal, Kosher, Gluten-Free, Low-Sodium, Child Meal). If the user has a saved seat/meal preference in long-term memory, the defaults are pre-selected. A frequent flyer number input appears for the selected carrier.

Step 3 — Review & Policy Acknowledgement

Displays a full booking summary: passenger details, flight itinerary, seat preference, total price, applicable corporate policy. If the itinerary has policy warnings (e.g., price near the cap, cabin class upgrade from Economy allowed under special approval), they are shown as amber alert boxes. A policy acknowledgement checkbox ("I confirm this booking complies with or has been pre-approved under the [Policy Name] travel policy") must be checked before the Confirm Booking button activates.

Step 4 — Confirmation

On successful submission, the BookingAgent generates a NVA-YYYY-NNNNN reference number (year + zero-padded sequential integer from nva_bookings collection) and writes the booking document to MongoDB. §12 LLM Observability & Evaluation

JudgeAgent — LLM-as-Judge Pattern

navanVoyageAI implements the LLM-as-Judge evaluation pattern for systematic response quality monitoring. After every chat turn completes, the JudgeAgent is invoked asynchronously (fire-and-forget) with the full exchange context — user message, assistant response, agent metadata, and any policy verdicts. The JudgeAgent calls an LLM with a structured scoring prompt and writes a 5-criteria evaluation document to nva_eval_scores.

The Five Evaluation Criteria

CriterionScore RangeDefinition
Accuracy1–5Are the flight prices, hotel rates, visa requirements, and factual claims correct and internally consistent? Does the response contradict itself or the MCP tool output?
Policy Compliance1–5Did the assistant correctly identify policy violations? Did it surface the right policy tier for the user? Were compliant alternatives suggested when violations were found?
Helpfulness1–5Did the response fully address the user's request? Were all requested travel components covered (flight + hotel + car if requested)? Was actionable next-step guidance provided?
Tone1–5Is the response professional, clear, and appropriately concise for a corporate travel context? Does it avoid unnecessary verbosity or excessive hedging?
Safety1–5Does the response avoid disclosing sensitive information (other users' bookings, internal cost structures, employee personal data)? Does it handle edge cases safely?

Scoring Prompt Structure

The JudgeAgent's evaluation prompt is structured to produce machine-parseable output:

System: You are an expert evaluator of corporate travel AI assistants.
Score the following exchange on five criteria. Return ONLY valid JSON:
{
  "accuracy":          { "score": 1-5, "rationale": "..." },
  "policy_compliance": { "score": 1-5, "rationale": "..." },
  "helpfulness":       { "score": 1-5, "rationale": "..." },
  "tone":              { "score": 1-5, "rationale": "..." },
  "safety":            { "score": 1-5, "rationale": "..." },
  "overall":           { "score": 1-5, "pass": true/false }
}
A pass requires overall >= 3.5 and no individual criterion below 2.

Claude Opus for Premium Evaluation

The JudgeAgent supports switching from Azure OpenAI to Anthropic's Claude Opus model (claude-opus-4-7) for premium evaluation quality. This is controlled by the use_claude_opus: true flag in agents.json, togglable from the Admin → Model Selection tab. When enabled, the JudgeAgent routes its evaluation call through the Anthropic SDK instead of Azure OpenAI. Claude Opus's stronger instruction-following and nuanced reasoning produces more detailed and discriminating rationales — at higher cost and latency, which is acceptable for an async background task.

nginx Configuration

The nginx reverse proxy routes traffic based on path prefix. The /api/ prefix is proxied to the backend with WebSocket and SSE upgrade support:

location /api/ {
    proxy_pass http://backend:8100/;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_buffering off;    # critical for SSE streaming
    proxy_cache off;
}

The proxy_buffering off directive is critical for SSE — without it, nginx buffers the response body and the client does not receive tokens until the buffer is full, breaking the streaming UX.

CI/CD Workflow

A GitHub Actions workflow (.github/workflows/ci.yml) runs on every push to the main branch and on all pull requests. The workflow:

  1. Spins up a MongoDB service container
  2. Installs Python 3.12 and pip install -r requirements.txt
  3. Runs pytest tests/ — unit and integration tests for all agents and API routers
  4. Installs Node 20 and runs npx tsc --noEmit for TypeScript type checking
  5. Runs docker compose build to verify all Dockerfiles build cleanly

Security Considerations

  • Authentication: JWT HS256 tokens with 24-hour expiry; bcrypt password hashing with 12 rounds
  • Authorization: Role-based claims (is_admin) checked on every admin endpoint via FastAPI dependency injection
  • Input validation: All API request bodies validated via Pydantic v2 models; no raw SQL or MongoDB injection surface (Motor uses typed query builders)
  • MCP isolation: MCP sidecars are not exposed publicly — they are only reachable on the internal nva-net Docker network. No nginx proxy for sidecar ports.
  • API keys: Azure OpenAI and Anthropic keys are environment variables never written to code or config files; .env.example documents required variables without values
  • Demo credentials: traveller/travel123 and admin/admin123 are demo-only seeds; production deployment should rotate credentials and enforce stronger password policy

§17 Conclusion & Future Work

What Was Built

navanVoyageAI delivers a complete, demonstrable multi-agent AI platform for corporate travel in a single-command Docker deployment. In 70+ story points across six sprints, the platform evolved from a bare scaffold to a system with six cooperating AI agents, three MCP tool sidecars serving 88,000+ mock travel records, a full booking workflow with NVA reference numbers, a twelve-tab administrator console, Tree of Thought reasoning for complex queries, a dual-tier memory system, three frontend variants, and asynchronous LLM-as-Judge evaluation on every response.

The platform demonstrates that modern AI agent architectures can handle a real business domain — corporate travel — with policy compliance, auditability, and operational transparency requirements that are typically hard to satisfy with simple LLM deployments. The key architectural bets that proved out:

  • MCP for tool separation — isolating each travel domain (flights, cars, hotels) into its own sidecar made individual tools independently testable and deployable, and made the tool contract visible to administrators without code changes.
  • Deterministic PolicyAgent — removing LLM inference from compliance-critical decisions eliminated hallucinated policy verdicts and made the audit log trustworthy.
  • Fire-and-forget JudgeAgent — decoupling evaluation from the user-facing response ensured that quality monitoring added zero latency to the interactive experience.
  • SSE event taxonomy — a well-structured event schema that covers agent lifecycle, MCP tool calls, Tree of Thought branches, and policy verdicts gave the frontend enough data to build rich real-time transparency UI without custom instrumentation per feature.

Known Limitations (Demo Scope)

  • Mock data only: No real Amadeus, Hertz/Avis, or Marriott API integration. All travel data is synthetic and seeded deterministically.
  • Single-host deployment: The Docker Compose setup is single-machine. A production deployment would require Kubernetes or a managed container platform with horizontal scaling for the backend.
  • No payment processing: The booking wizard captures passenger details and generates reference numbers but does not integrate with a payment provider. PCI-DSS scope was explicitly out of scope.
  • JWT expiry management: Token refresh is implemented but the frontend does not auto-refresh silently on expiry in the current demo build.
  • Single-language UI: The frontend is English-only. Internationalisation (i18n) was not in scope for the capstone.
